# Шаг 10. Тестирование, проверка сценариев и настройка качества сигналов

## 1. Цель шага

- Зафиксировать, как проверять корректность работы бота до и после реализации.
- Определить набор тестовых сценариев для аналитики, Telegram-интерфейса, хранения состояния и интеграций.
- Описать, как оценивать и улучшать качество сигналов.

## 2. Общий принцип проверки

- Тестирование должно охватывать не только код, но и пользовательские сценарии.
- Для бота важны три вида качества: корректность логики, устойчивость работы и полезность сигналов.
- Проверка качества сигналов не может опираться только на отсутствие ошибок — нужна оценка аналитического результата.

## 3. Основные направления тестирования

Для первой версии выделяются:

- модульное тестирование;
- интеграционное тестирование;
- сценарное тестирование Telegram-потоков;
- проверка устойчивости хранения и восстановления;
- проверка качества сигналов на истории;
- ручная экспертная валидация отдельных кейсов.

## 4. Модульное тестирование

### 4.1 Что тестировать модульно

- нормализацию свечей;
- маппинг тикеров;
- поиск экстремумов;
- построение кандидатов P0-P5;
- раннюю структурную фильтрацию;
- валидацию пропорций и Фибоначчи;
- антидубли;
- форматирование уведомлений;
- чтение и запись состояния.

### 4.2 Цель модульных тестов

- быстро находить поломки в логике;
- контролировать регрессии при изменении правил;
- изолированно проверять ключевые расчеты.

## 5. Интеграционное тестирование

Интеграционно должны проверяться:

- связка провайдера списка монет и маппинга пар;
- связка провайдера свечей и нормализации;
- связка движка поиска и валидации Фибоначчи;
- связка Notification Service и Chart Rendering Service;
- связка Persistence Service и восстановления состояния;
- связка Telegram-сценариев с сервисным слоем.

Цель:

- убедиться, что модули правильно взаимодействуют между собой;
- найти ошибки на границах между слоями.

## 6. Сценарное тестирование Telegram

Обязательно нужно проверить сценарии:

- первый запуск через `/start`;
- запуск мониторинга;
- остановку мониторинга;
- просмотр списка пар;
- добавление пары без указания таймфрейма;
- добавление пары с указанным таймфреймом;
- изменение таймфрейма;
- удаление пары;
- ручную проверку пары;
- изменение настроек;
- просмотр статуса.

Особые проверки:

- после любого сценария Reply Keyboard должна сохраняться;
- при ошибке пользователь должен получать понятный ответ;
- при ручной проверке бот не должен молчать, даже если структура не найдена.

## 7. Тестирование аналитики на истории

Для оценки аналитики нужен отдельный набор исторических кейсов:

- кейсы с хорошо выраженной классической пятиволновкой;
- кейсы с похожими, но некорректными структурами;
- кейсы с сильным рыночным шумом;
- кейсы с недостаточным числом свечей;
- кейсы для long и short;
- кейсы с диагоналями, если они допущены спецификацией.

## 8. Категории тестовых датасетов

Для первой версии полезно сформировать:

- `positive_cases` — ожидается корректное распознавание;
- `negative_cases` — ожидается отказ;
- `borderline_cases` — неоднозначные случаи;
- `api_error_cases` — ошибки внешних источников;
- `state_recovery_cases` — восстановление после перезапуска.

## 9. Проверка качества сигналов

Качество сигналов оценивается не только по числу найденных структур, но и по их полезности.

Для первой версии вводятся такие показатели:

- доля подтвержденных сигналов среди отправленных;
- доля ложноположительных сигналов;
- доля пропущенных очевидных структур;
- доля слабых `probable`, которые не стоило бы отправлять;
- число подавленных дублей;
- стабильность результата при повторной проверке одного и того же участка данных.

## 10. Практический процесс настройки качества

Цикл настройки качества должен быть таким:

1. Выбрать исторический кейс.
2. Прогнать аналитический конвейер.
3. Сравнить результат с ожидаемым экспертным сценарием.
4. Зафиксировать тип ошибки.
5. Подстроить чувствительность, фильтрацию или пороги.
6. Повторно прогнать набор контрольных кейсов.

Принцип:

- нельзя улучшать один кейс ценой разрушения базового поведения на остальных.

## 11. Типы аналитических ошибок

При настройке качества нужно различать:

- `false_positive` — найдена структура там, где ее нет;
- `false_negative` — не найдена структура там, где она есть;
- `weak_confirmation` — структура найдена, но статус завышен;
- `duplicate_noise` — дубли или почти дубли ушли пользователю;
- `unstable_result` — один и тот же участок дает разный итог при повторных проверках.

## 12. Что именно подстраивается при улучшении качества

Для первой версии можно регулировать:

- чувствительность поиска экстремумов;
- минимальное расстояние между экстремумами;
- глубину истории;
- режим поиска `strict`, `standard`, `aggressive`;
- диапазоны допустимых пропорций;
- правила ранжирования `confirmed` и `probable`;
- правила отправки `probable` в фоне.

## 13. Контроль регрессий

После каждого изменения аналитики нужно повторно проверять:

- набор положительных кейсов;
- набор отрицательных кейсов;
- ручную проверку пары;
- антидубли;
- генерацию уведомления;
- восстановление состояния после перезапуска.

Если изменение улучшило один класс кейсов, но ухудшило базовые сценарии:

- оно не должно считаться принятым без дополнительной корректировки.

## 14. Проверка устойчивости

Нужно отдельно проверять:

- отсутствие сети;
- частичные ошибки API;
- превышение rate limit;
- пустые ответы провайдера;
- поврежденное состояние в хранилище;
- ошибку рендера графика;
- ошибку отправки сообщения в Telegram;
- перезапуск контейнера Docker.

Ожидаемое поведение:

- бот не должен падать без понятного состояния;
- должен сохраняться полезный журнал;
- пользователь не должен получать поток однотипных ошибок.

## 15. Проверка хранения и восстановления

Обязательно нужно проверить:

- сохранение настроек;
- сохранение watchlist;
- сохранение индивидуальных таймфреймов;
- сохранение истории сигналов;
- корректное восстановление после остановки;
- корректное восстановление после аварийного перезапуска;
- работу того же сценария при запуске через Docker volume.

## 16. Проверка качества уведомлений

Нужно проверять:

- читаемость графика на телефоне;
- корректность подписей волн;
- корректность пары и таймфрейма;
- отсутствие дублей;
- явный ответ при ручной проверке без найденной структуры;
- fallback на текст при ошибке визуализации.

## 17. Ручная экспертная валидация

Даже при наличии автоматических тестов часть кейсов должна проверяться вручную:

- визуально ли структура действительно похожа на классический импульс;
- не завышен ли статус `confirmed`;
- не пропущены ли очевидные структурные нарушения;
- не слишком ли много шума уходит пользователю.

## 18. Минимальный выпускной чек для первой версии

Перед рабочим использованием должны быть подтверждены:

- базовые Telegram-сценарии проходят без тупиков;
- дефолтный таймфрейм `5m` работает корректно;
- данные после перезапуска не теряются;
- антидубли работают;
- графики отправляются корректно;
- ручная проверка возвращает понятный ответ;
- набор контрольных исторических кейсов не деградировал.

## 19. Docker-совместимость тестирования

- Часть проверок должна запускаться локально и в контейнере.
- Поведение в Docker не должно отличаться от локального при одинаковых входных данных.
- Особенно важно проверить storage path, volume и генерацию графиков в контейнерной среде.

## 20. Итоговые критерии качества первой версии

Первая версия считается пригодной, если:

- бот стабильно проходит основные пользовательские сценарии;
- аналитика не генерирует очевидный поток ложных сигналов;
- пользователь получает понятные ответы и статусы;
- состояние и история переживают перезапуск;
- ключевые ошибки диагностируются через журналы.

## 21. Результат шага 10

- Зафиксирована стратегия тестирования и проверки качества системы.
- Определены тестовые направления для аналитики, Telegram, хранения и Docker.
- Описан процесс улучшения качества сигналов и контроля регрессий.
- Базовый цикл проектной спецификации завершен и может использоваться как основа для перехода к реализации.
