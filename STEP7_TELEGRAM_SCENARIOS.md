# Шаг 7. Telegram-сценарии управления ботом

## 1. Цель шага

- Зафиксировать пользовательские сценарии работы с ботом в Telegram.
- Определить структуру меню, переходов и ответов бота.
- Разделить главное управление, настройки, ручные проверки и сервисные уведомления.

## 2. Основной принцип интерфейса

- Главное управление строится через Reply Keyboard.
- Основные действия всегда доступны в нижней клавиатуре, без необходимости искать старые сообщения.
- Inline-кнопки могут использоваться только для локальных одноразовых действий внутри конкретного сценария, если это действительно упрощает ввод.
- Основные пользовательские сценарии не должны зависеть от Inline-клавиатур.

## 3. Главное меню

Главное меню должно содержать:

- `Старт` или `Стоп`
- `Список пар`
- `Добавить пару`
- `Изменить таймфрейм`
- `Удалить пару`
- `Проверить пару`
- `Настройки`
- `Статус`

Правила:

- После запуска бота кнопка `Старт` заменяется на `Стоп`.
- После остановки мониторинга кнопка `Стоп` заменяется обратно на `Старт`.
- Reply Keyboard должна оставаться доступной после завершения любого основного сценария.

## 4. Сценарий первого входа

### 4.1 Команда `/start`

Бот должен:

- приветствовать пользователя;
- кратко объяснить назначение бота;
- показать текущее состояние мониторинга;
- вывести Reply Keyboard главного меню.

### 4.2 Сообщение первого запуска

Сообщение должно включать:

- что бот ищет классическую 5-волновую структуру Эллиотта;
- что дефолтный таймфрейм старта равен `5m`;
- что список пар может быть сформирован автоматически или вручную;
- как перейти к проверке пары и настройкам.

## 5. Сценарий запуска мониторинга

### 5.1 Нажатие `Старт`

Бот выполняет:

1. Проверяет текущие настройки.
2. Берет топ монет из рыночного источника.
3. Фильтрует неподходящие активы.
4. Маппит активы в торговые пары.
5. Формирует watchlist.
6. Запускает фоновый мониторинг.

### 5.2 Ответ пользователю при старте

Бот должен сообщить:

- что мониторинг запущен;
- сколько пар добавлено в отслеживание;
- какая биржа используется для свечей;
- какой дефолтный таймфрейм применяется для новых пар: `5m`;
- какой интервал сканирования активен.

### 5.3 Ошибки старта

Если старт не удался, бот должен вернуть понятную причину:

- недоступен источник рынка;
- не удалось получить пары;
- отсутствуют необходимые настройки;
- ошибка внешнего API.

## 6. Сценарий остановки мониторинга

### 6.1 Нажатие `Стоп`

Бот выполняет:

- останавливает новые циклы мониторинга;
- сохраняет текущее состояние;
- не удаляет watchlist, настройки и историю сигналов.

### 6.2 Ответ пользователю при остановке

Бот должен сообщить:

- что мониторинг остановлен;
- что список пар и настройки сохранены;
- что запуск можно продолжить позднее без повторной настройки.

## 7. Сценарий `Список пар`

Бот должен показывать:

- все активные пары;
- таймфрейм каждой пары;
- источник добавления: auto или manual;
- краткий статус: активна, приостановлена, недавно проверялась.

Если список пуст:

- бот должен явно написать, что пары для отслеживания пока отсутствуют.

## 8. Сценарий `Добавить пару`

### 8.1 Последовательность

Бот должен:

1. Запросить у пользователя пару.
2. При необходимости уточнить таймфрейм.
3. Если таймфрейм не указан, использовать `5m`.
4. Проверить доступность пары у провайдера свечей.
5. Добавить пару в watchlist.

### 8.2 Ответ после добавления

Бот должен сообщить:

- что пара добавлена;
- какой таймфрейм установлен;
- будет ли она участвовать в фоновом мониторинге сразу.

### 8.3 Ошибки

Бот должен уметь объяснить:

- пара не существует на выбранной бирже;
- таймфрейм не поддерживается;
- пара уже есть в списке.

## 9. Сценарий `Изменить таймфрейм`

### 9.1 Последовательность

Бот должен:

1. Показать пары, для которых можно изменить таймфрейм.
2. Получить выбранную пару.
3. Получить новый таймфрейм.
4. Проверить поддерживаемость таймфрейма.
5. Сохранить новый `PairMonitoringConfig`.

### 9.2 Ответ

Бот должен сообщить:

- что таймфрейм обновлен;
- какое новое значение применяется;
- с какого момента оно будет участвовать в мониторинге.

## 10. Сценарий `Удалить пару`

### 10.1 Последовательность

Бот должен:

1. Показать список активных пар.
2. Получить выбор пользователя.
3. Удалить пару из активного мониторинга.
4. Сохранить историю сигналов и служебные записи.

### 10.2 Ответ

Бот должен сообщить:

- что пара удалена из мониторинга;
- что исторические сигналы по ней сохранены.

## 11. Сценарий `Проверить пару`

### 11.1 Последовательность

Бот должен:

1. Запросить пару.
2. Запросить таймфрейм или использовать `5m`, если пользователь его не указал.
3. Выполнить разовый анализ через общий аналитический конвейер.
4. Вернуть лучший найденный сценарий или понятную причину отказа.

### 11.2 Ответ при успешной проверке

Бот должен вернуть:

- пару и таймфрейм;
- направление: long или short;
- статус: `confirmed` или `probable`;
- краткое описание найденной структуры;
- график с волнами;
- краткую диагностическую сводку.

### 11.3 Ответ при отсутствии структуры

Бот должен вернуть понятный ответ, а не молчать.

Минимальный формат:

- структура не подтверждена;
- если возможно, краткая причина: недостаточно данных, слабые пропорции, нарушена структура или нет подходящего кандидата.

## 12. Сценарий `Настройки`

Раздел настроек должен позволять менять:

- интервал сканирования;
- таймфрейм по умолчанию;
- режим поиска: `strict`, `standard`, `aggressive`;
- количество автоматически добавляемых пар;
- биржу свечей, если эта опция будет поддерживаться;
- объяснение причин отклонения для ручной проверки;
- режим отправки вероятных сигналов в будущем.

### 12.1 Правила для таймфрейма по умолчанию

- Начальное значение при старте проекта: `5m`.
- Новое значение не должно менять уже явно настроенные пары без отдельного действия пользователя.
- Новое значение применяется к новым парам и тем сценариям, где таймфрейм не указан явно.

## 13. Сценарий `Статус`

Бот должен показывать:

- запущен ли мониторинг;
- сколько пар сейчас отслеживается;
- какой дефолтный таймфрейм активен;
- какой интервал сканирования используется;
- какая биржа подключена;
- когда был последний цикл проверки;
- была ли последняя ошибка.

## 14. Формат автоматических уведомлений о сигнале

Сообщение о найденной структуре должно содержать:

- пару;
- таймфрейм;
- направление;
- статус сигнала;
- краткое описание типа структуры;
- основные подтверждения;
- график с разметкой волн.

### 14.1 Ограничение шума

- Повторные сигналы по одной и той же структуре не отправляются.
- Слабые или отклоненные сценарии в фоне не отправляются.
- Служебные ошибки мониторинга не должны спамить пользователю в каждом цикле.

## 15. Служебные сообщения и ошибки

Бот должен уметь отправлять понятные сервисные ответы:

- источник рынка временно недоступен;
- биржа не вернула свечи;
- превышен лимит запросов;
- конфигурация неполная;
- пара не поддерживается;
- таймфрейм не поддерживается.

Принцип:

- текст ошибки должен быть человеческим;
- при этом внутри системы должна сохраняться машинная причина сбоя.

## 16. Состояния сценариев

Для интерфейса Telegram нужно учитывать промежуточные состояния:

- ожидание ввода пары;
- ожидание ввода таймфрейма;
- ожидание подтверждения удаления;
- ожидание нового значения настройки;
- завершенный сценарий;
- ошибка сценария;
- отмененный сценарий.

## 17. Правила возврата в главное меню

- После завершения любого пользовательского сценария бот возвращает Reply Keyboard главного меню.
- После ошибки бот тоже должен возвращать пользователя в понятное состояние.
- Для многошаговых сценариев должна быть возможность отмены и возврата в меню.

## 18. Docker-совместимость

- Telegram-сценарии не зависят от Docker напрямую.
- При контейнерном запуске логика сценариев должна работать одинаково.
- Все токены и настройки Telegram передаются через переменные окружения.

## 19. Результат шага 7

- Зафиксированы основные пользовательские сценарии Telegram-бота.
- Главное управление закреплено через Reply Keyboard как основной интерфейс навигации.
- Описаны ответы при старте, остановке, проверке пары, настройках, статусе и уведомлениях.
- Следующий шаг можно посвятить уведомлениям, визуализации и формату отправки графиков.
