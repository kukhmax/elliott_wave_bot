# Шаг 9. Хранение состояния и журналирование

## 1. Цель шага

- Зафиксировать, какие данные бот сохраняет между перезапусками.
- Определить, какие журналы нужны для стабильной работы и отладки.
- Подготовить проект к надежной работе в локальном режиме и через Docker.

## 2. Основной принцип хранения

- Состояние должно сохраняться так, чтобы остановка или перезапуск бота не уничтожали рабочий контекст.
- Постоянно хранятся только значимые данные.
- Временные аналитические промежуточные данные не должны перегружать хранилище.
- Журналирование должно быть полезным для диагностики, но не превращаться в поток шума.

## 3. Что считается постоянным состоянием

Для первой версии постоянно сохраняются:

- глобальные настройки приложения;
- watchlist;
- конфигурация мониторинга по каждой паре;
- история сигналов;
- последнее служебное состояние бота;
- журнал ошибок и ключевых событий в ограниченном объеме.

## 4. Что не нужно хранить постоянно

Для первой версии не требуется постоянное хранение:

- полного архива свечей;
- всех промежуточных экстремумов;
- всех кандидатных структур, отброшенных на раннем этапе;
- временных файлов графиков после отправки, если они не нужны для отладки.

## 5. Логические хранилища первой версии

### 5.1 Settings Store

Хранит:

- `AppSettings`

Назначение:

- восстанавливать настройки при перезапуске;
- поддерживать изменение настроек через Telegram;
- передавать параметры всем слоям системы.

### 5.2 Watchlist Store

Хранит:

- список `TradingPair`
- `PairMonitoringConfig`

Назначение:

- сохранять пары между сессиями;
- не терять вручную добавленные пары;
- помнить индивидуальные таймфреймы и параметры сканирования.

### 5.3 Signal History Store

Хранит:

- `SignalRecord`
- краткую информацию о последнем отправленном сигнале по паре

Назначение:

- антидубли;
- история принятых решений;
- объяснение, почему сигнал был отправлен или подавлен.

### 5.4 Runtime State Store

Хранит:

- сокращенный `RuntimeState`

Назначение:

- понимать, был ли мониторинг активен;
- восстанавливать понятное состояние после перезапуска;
- показывать пользователю актуальный статус.

### 5.5 Event Log Store

Хранит:

- ключевые служебные события;
- ошибки интеграций;
- изменения настроек;
- запуск и остановку мониторинга.

Назначение:

- диагностика;
- аудит поведения бота;
- разбор нестабильной работы.

## 6. Минимальный формат хранения первой версии

Для первой версии допускается файловое хранение.

Минимальный набор файлов или логических объектов:

- `settings`
- `watchlist`
- `signal_history`
- `runtime_state`
- `event_log`

Принцип:

- структура должна быть простой;
- файлы должны быть читаемыми и восстанавливаемыми;
- формат должен подходить для внешнего volume в Docker.

## 7. Требования к файловому хранению

- Хранилище должно иметь единый `storage_path`.
- Все данные должны сохраняться атомарно или максимально близко к этому.
- Повреждение одного файла не должно ломать всё приложение без понятной ошибки.
- Должен быть предусмотрен безопасный сценарий чтения пустого или отсутствующего файла.

## 8. Поведение при старте приложения

При запуске бот должен:

1. Прочитать `AppSettings`.
2. Прочитать watchlist и конфигурации пар.
3. Прочитать историю сигналов.
4. Прочитать последнее runtime-состояние.
5. Инициализировать внутренние сервисы из восстановленных данных.

Если часть хранилища отсутствует:

- используется безопасное значение по умолчанию;
- ошибка логируется как событие восстановления;
- бот продолжает работу, если это возможно.

## 9. Поведение при остановке приложения

При штатной остановке бот должен:

- сохранить текущее runtime-состояние;
- завершить запись последних служебных событий;
- не терять накопленные настройки и watchlist;
- завершить открытые операции записи корректно.

## 10. Поведение при сбое

Если произошел сбой:

- последняя критичная ошибка должна быть записана в журнал;
- при следующем старте бот должен уметь восстановить хотя бы базовые настройки и watchlist;
- поврежденное состояние должно выявляться и логироваться;
- пользователь не должен получать бесконечный поток одинаковых ошибок.

## 11. Сценарий восстановления после перезапуска

После перезапуска:

- бот восстанавливает настройки;
- бот восстанавливает список пар и их таймфреймы;
- бот восстанавливает историю сигналов для антидублей;
- статус мониторинга может быть показан как `stopped` или `paused`, пока пользователь явно не запустит мониторинг снова, если не задан иной режим автозапуска;
- состояние должно быть совместимо с запуском из Docker-контейнера после пересоздания контейнера.

## 12. Журналирование: уровни

Для первой версии достаточно четырех уровней:

- `INFO`
- `WARNING`
- `ERROR`
- `CRITICAL`

Принцип:

- `INFO` — обычные значимые события;
- `WARNING` — ситуации, не ломающие работу, но требующие внимания;
- `ERROR` — ошибка конкретной операции;
- `CRITICAL` — сбой, влияющий на устойчивость всего приложения.

## 13. Что обязательно логировать

Обязательно логируются:

- старт приложения;
- остановка приложения;
- запуск мониторинга;
- остановка мониторинга;
- загрузка или восстановление состояния;
- ошибки чтения и записи хранилища;
- ошибки внешних API;
- изменение настроек;
- добавление, удаление и изменение пары;
- отправка сигнала;
- подавление сигнала как дубля;
- ошибка построения графика;
- ошибка отправки в Telegram.

## 14. Что не нужно логировать избыточно

Не нужно превращать в спам:

- каждую внутреннюю мелкую проверку кандидата;
- каждую свечу;
- однотипные повторяющиеся временные ошибки без изменения контекста;
- секреты, токены, API-ключи и чувствительные параметры.

## 15. Формат записи событий

Каждое событие журнала должно содержать:

- время;
- уровень;
- модуль или сервис;
- тип события;
- краткое описание;
- при необходимости машинный код причины;
- минимальный полезный контекст.

Пример контекста:

- символ;
- таймфрейм;
- имя провайдера;
- идентификатор сигнала;
- категория ошибки.

## 16. Отдельные категории журналов

Для удобства логически выделяются:

- `system_events`
- `market_data_events`
- `analysis_events`
- `notification_events`
- `storage_events`

Это не обязательно разные файлы в первой версии, но это должны быть разные категории событий.

## 17. Журнал решений по сигналам

Отдельно нужно фиксировать:

- какой кандидат был выбран лучшим;
- какой статус он получил;
- почему сигнал был отправлен;
- почему сигнал был подавлен;
- почему структура была отклонена при ручной проверке.

Это особенно важно для:

- отладки ложных сигналов;
- контроля антидублей;
- последующего улучшения аналитики.

## 18. Политика хранения логов первой версии

Для первой версии:

- журнал не должен расти бесконечно;
- должна быть простая политика ограничения размера или количества записей;
- старые служебные события могут очищаться или ротироваться;
- история сигналов очищается осторожнее, чем технические ошибки.

## 19. Работа с Docker

- Все постоянные данные должны храниться по пути, совместимому с внешним volume.
- Логи должны быть доступны как минимум через стандартный вывод контейнера.
- При необходимости часть логов может дублироваться в файловое хранилище.
- Пересоздание контейнера не должно уничтожать настройки, watchlist и историю сигналов.

## 20. Минимальные требования к надежности

- отсутствие хранилища не должно приводить к неуправляемому падению без сообщения;
- чтение поврежденных данных должно завершаться контролируемой ошибкой;
- запись состояния не должна ломать рабочий цикл мониторинга целиком;
- важные ошибки должны быть доступны для диагностики после перезапуска.

## 21. Границы ответственности

Persistence Service отвечает за:

- чтение и запись постоянного состояния;
- восстановление данных при запуске;
- безопасную работу с `storage_path`.

Logging Service отвечает за:

- запись событий и ошибок;
- разграничение уровней логирования;
- защиту от избыточного шума;
- недопущение записи секретов.

Monitoring Coordinator и остальные сервисы отвечают за:

- передачу корректных событий и состояний в Persistence Service и Logging Service.

## 22. Результат шага 9

- Зафиксированы правила постоянного хранения состояния.
- Определены основные типы журналов, уровни логирования и обязательные события.
- Описано восстановление после перезапуска и требования к Docker-volume.
- Следующий шаг можно посвятить тестированию, проверке сценариев и настройке качества сигналов.
