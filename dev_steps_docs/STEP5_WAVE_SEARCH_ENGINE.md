# Шаг 5. Базовый движок поиска волн

## 1. Цель шага

- Зафиксировать, как бот находит кандидатов на классическую 5-волновую структуру.
- Разделить поиск кандидатов, базовую структурную проверку и передачу в строгую валидацию.
- Определить минимальный алгоритм первой версии без избыточной сложности.

## 2. Роль движка в общей архитектуре

- Движок работает после слоя получения и нормализации свечей.
- На вход он получает `MarketSeries` по конкретной паре и таймфрейму.
- Если для пары не задан индивидуальный таймфрейм, используется стартовый `default_timeframe = 5m`.
- Движок не отправляет сообщения и не работает с Telegram напрямую.
- Движок не выбирает лучшие пары рынка, а анализирует только уже переданные ему данные.

## 3. Что считается задачей шага 5

- Найти локальные экстремумы в серии свечей.
- Построить из них кандидатные последовательности P0-P5.
- Проверить базовую направленность и очередность волн 1-2-3-4-5.
- Отбросить явно некорректные сценарии до углубленной валидации.
- Передать ограниченный список лучших кандидатов в Elliott Validation Service.

## 4. Что не входит в шаг 5

- Точная оценка по Фибоначчи как главный критерий качества.
- Финальный scoring с полной системой весов.
- Оптимизация производительности под большие объемы данных.
- Внутреннее полное разложение всех подволн.

## 5. Входные данные движка

Движок получает:

- `symbol`
- `timeframe`
- `bars`
- `history_depth`
- настройки чувствительности экстремумов
- режим поиска

Предварительные условия:

- Свечи уже нормализованы.
- Свечи отсортированы по времени.
- Неполная текущая свеча исключена из анализа.
- Истории достаточно для построения минимум 6 опорных точек.

## 6. Общий конвейер поиска

### 6.1 Этап 1. Предобработка свечей

- Проверить минимальный объем данных.
- Удалить неполную последнюю свечу, если она не закрыта.
- Подготовить удобные массивы цен и индексов для дальнейшего анализа.

### 6.2 Этап 2. Поиск локальных экстремумов

- Выделить локальные максимумы и минимумы.
- Отфильтровать слишком слабые колебания, которые являются рыночным шумом.
- Для каждого экстремума сохранить индекс, время, цену, тип и силу.

### 6.3 Этап 3. Формирование свингов

- Преобразовать экстремумы в чередующуюся последовательность максимумов и минимумов.
- Удалить подряд идущие однотипные экстремумы, оставив более значимый.
- Сформировать очищенную swing-последовательность для перебора кандидатов.

### 6.4 Этап 4. Построение кандидатов P0-P5

- Перебирать окна из шести опорных точек.
- Для каждого окна строить long- и short-интерпретацию, если это не противоречит направлению.
- Проверять, что точки действительно образуют чередование волн 1-2-3-4-5.

### 6.5 Этап 5. Базовый структурный фильтр

- Проверять направленность волн.
- Проверять, что волна 2 идет против волны 1.
- Проверять, что волна 3 продолжает тренд после волны 2.
- Проверять, что волна 4 корректирует волну 3.
- Проверять, что волна 5 снова идет по направлению основного движения.

### 6.6 Этап 6. Жесткая ранняя отсечка

- Сразу отклонять кандидатов с нарушением базовых правил Эллиотта.
- Не передавать в глубокую валидацию заведомо некорректные сценарии.

### 6.7 Этап 7. Сокращение числа кандидатов

- Удалять почти одинаковые кандидаты с одними и теми же ключевыми точками.
- Оставлять ограниченное число самых структурно чистых вариантов.

### 6.8 Этап 8. Передача в Elliott Validation Service

- Передавать только те сценарии, которые прошли базовую фильтрацию.
- Вместе со сценарием передавать промежуточные диагностические признаки.

## 7. Правила поиска локальных экстремумов

Для первой версии вводятся простые требования:

- Экстремум должен быть заметен относительно соседних свечей.
- Между соседними экстремумами должно быть минимальное расстояние по числу свечей.
- Слишком мелкие колебания не считаются отдельными волнами.
- Один и тот же бар не может одновременно быть двумя разными опорными точками одного кандидата.

Практическая цель:

- не дробить рынок на слишком мелкие движения;
- не пропускать достаточно выраженные swing-участки;
- обеспечить повторяемый результат для одного и того же участка данных.

## 8. Правила формирования кандидата P0-P5

Для long-кандидата:

- `P1 > P0`
- `P2 < P1` и `P2 > P0`
- `P3 > P1`
- `P4 < P3`
- `P5 > P3`

Для short-кандидата:

- те же условия в зеркальном виде

Дополнительные требования:

- точки должны идти строго слева направо по времени;
- между соседними точками должно быть измеримое расстояние;
- кандидат не должен состоять из почти нулевых волн;
- волны не должны схлопываться в один горизонтальный участок.

## 9. Базовые правила ранней структурной проверки

На этапе движка проверяются следующие правила:

- волна 2 не заходит за начало волны 1;
- волна 3 обновляет экстремум волны 1 по направлению тренда;
- волна 4 корректирует волну 3, а не продолжает ее;
- волна 5 обновляет экстремум волны 3 по направлению тренда;
- волны 1, 3 и 5 направлены в одну сторону;
- волны 2 и 4 направлены против основного движения;
- волна 3 не должна быть самой короткой среди 1, 3 и 5;
- для обычного импульса волна 4 не заходит в ценовую зону волны 1.

Примечание:

- Исключения для диагоналей допускаются, но окончательная интерпретация диагонали закрепляется в Elliott Validation Service.

## 10. Представление кандидата в движке

После построения кандидат должен содержать:

- идентификатор кандидата;
- `symbol`;
- `timeframe`;
- направление;
- точки `P0-P5`;
- длины волн 1-5;
- список использованных экстремумов;
- краткий набор ранних структурных признаков;
- причины отклонения, если кандидат отброшен на раннем этапе.

## 11. Режимы поиска

Для первой версии фиксируются три режима:

### 11.1 Strict

- высокий порог силы экстремума;
- меньшая терпимость к шуму;
- меньше кандидатов, но выше чистота структуры.

### 11.2 Standard

- базовый режим по умолчанию;
- баланс между полнотой и качеством;
- рекомендуется для непрерывного мониторинга.

### 11.3 Aggressive

- более слабый порог экстремумов;
- больше кандидатов для дальнейшей валидации;
- выше риск ложных сценариев.

## 12. Ограничение комбинаторного взрыва

Чтобы движок не перебирал слишком много вариантов, вводятся ограничения:

- анализируется только ограниченное окно последних свечей;
- экстремумы предварительно очищаются;
- кандидаты строятся только из чередующейся swing-последовательности;
- почти одинаковые окна объединяются;
- число передаваемых в глубокую валидацию кандидатов ограничивается.

## 13. Причины раннего отклонения кандидата

Кандидат может быть отклонен по причинам:

- `not_enough_extremums`
- `invalid_wave_direction`
- `wave2_breaks_wave1_origin`
- `wave3_no_breakout`
- `wave4_overlap_wave1`
- `wave5_no_breakout`
- `wave3_shortest`
- `too_small_structure`
- `duplicate_candidate`

## 14. Выход движка

Результатом работы движка является:

- список `WaveCandidate`, прошедших базовую фильтрацию;
- список отклоненных кандидатов с причинами;
- метаданные анализа: число найденных экстремумов, число построенных окон, число допущенных кандидатов.

Если допустимых кандидатов нет:

- Elliott Validation Service не вызывается;
- для ручной проверки можно вернуть причину отсутствия валидных структур;
- для фонового мониторинга просто завершается цикл по этой паре.

## 15. Поведение при разных режимах запуска

### 15.1 Фоновый мониторинг

- Использует режим `standard`, если пользователь не настроил иначе.
- Для новых пар после старта применяется дефолтный таймфрейм `5m`, если не задано иное.
- Возвращает ограниченный набор кандидатов для следующего слоя.

### 15.2 Ручная проверка

- Использует таймфрейм, переданный пользователем.
- При отсутствии указания может использовать текущий дефолт приложения `5m`.
- Возвращает больше диагностической информации о причинах отклонения.

## 16. Границы ответственности относительно шага 6

Шаг 5 отвечает за:

- экстремумы;
- свинги;
- построение кандидатов;
- раннюю структурную отсечку.

Шаг 6 будет отвечать за:

- пропорции Фибоначчи;
- более точную оценку достоверности;
- ранжирование похожих сценариев;
- финальный статус `confirmed` или `probable`.

## 17. Docker-совместимость

- Движок не зависит от Docker напрямую.
- Параметры режима поиска, глубины истории и чувствительности должны читаться из конфигурации приложения.
- При контейнерном запуске поведение движка должно оставаться идентичным локальному запуску.

## 18. Результат шага 5

- Зафиксирован базовый алгоритм поиска кандидатных 5-волновых структур.
- Определены этапы: экстремумы, свинги, окна P0-P5, ранняя структурная фильтрация и передача валидации.
- Зафиксированы причины раннего отклонения и способы ограничения числа кандидатов.
- Следующий шаг можно посвятить отдельной валидации пропорций и Фибоначчи на уже подготовленных кандидатах.
