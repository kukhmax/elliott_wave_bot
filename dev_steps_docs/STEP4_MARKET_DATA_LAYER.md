# Шаг 4. Слой получения рыночных данных

## 1. Цель шага

- Зафиксировать, как бот получает список монет и свечные данные.
- Разделить источник рыночной вселенной и источник OHLCV.
- Определить правила маппинга тикеров, обработки ошибок API и лимитов.

## 2. Базовый принцип слоя данных

- Выбор монет и получение свечей считаются разными задачами.
- Для выбора монет используется отдельный провайдер рыночной вселенной.
- Для свечей используется отдельный провайдер рыночных данных.
- Внутри приложения данные всегда приводятся к единому нормализованному формату.
- Аналитический слой не должен знать детали конкретного API.

## 3. Принятое допущение для первой версии

- Источник списка топовых монет: CoinMarketCap.
- Источник свечей: одна основная криптобиржа с OHLCV API.
- Для первой версии в качестве базового провайдера свечей принимается Binance Spot как наиболее практичный и распространенный вариант.
- Все интеграции проектируются через абстракции, чтобы провайдер свечей можно было заменить позже без изменения доменной логики.

Причина такого выбора:

- CoinMarketCap хорошо подходит для ранжирования активов по market cap.
- Биржевой OHLCV API лучше подходит для стабильной загрузки свечей, чем агрегатор монет.
- Разделение источников соответствует уже выбранной архитектуре и уменьшает связность.

## 4. Основные компоненты слоя данных

### 4.1 Market Universe Provider

Назначение:

- Получать список топовых монет по market cap.
- Возвращать данные для отбора инструментов мониторинга.

Обязанности:

- Запрашивать рейтинг активов.
- Исключать стейблкоины.
- Ограничивать выбор числом активов из настроек.
- Возвращать список канонических активов без привязки к конкретной паре биржи.

Выходной результат:

- Список активов вида `BTC`, `ETH`, `SOL`.

### 4.2 Exchange Market Data Provider

Назначение:

- Загружать свечи и проверять доступность торговой пары на конкретной бирже.

Обязанности:

- Проверять наличие пары на бирже.
- Загружать OHLCV по паре и таймфрейму.
- Отдавать свечи в нормализованном виде.
- Возвращать понятные причины ошибок.

Выходной результат:

- Нормализованный `MarketSeries`.

### 4.3 Symbol Mapping Service

Назначение:

- Преобразовывать актив из списка market cap в конкретную биржевую пару.

Обязанности:

- Получать канонический актив, например `BTC`.
- Формировать целевую торговую пару для биржи, например `BTCUSDT`.
- Проверять, что такая пара реально доступна у провайдера свечей.
- Пропускать актив, если подходящая пара на бирже отсутствует.

### 4.4 Data Normalization Layer

Назначение:

- Приводить ответы всех провайдеров к единой внутренней модели.

Обязанности:

- Нормализовать поля свечи.
- Выравнивать таймстемпы.
- Приводить символы и таймфреймы к каноническому виду.
- Скрывать различия между провайдерами от аналитического слоя.

## 5. Целевой поток получения инструментов при старте

1. Monitoring Coordinator вызывает Market Universe Provider.
2. Market Universe Provider получает топ активов по market cap.
3. Фильтр исключает стейблкоины и неподходящие монеты.
4. Symbol Mapping Service пытается сопоставить каждый актив с торговой парой на выбранной бирже.
5. Пары, найденные на бирже, передаются в Watchlist Service.
6. Активы без доступной пары пропускаются и логируются как несопоставленные.

## 6. Целевой поток получения свечей для анализа

1. Monitoring Coordinator передает символ, таймфрейм и глубину истории.
2. Exchange Market Data Provider проверяет валидность пары.
3. Провайдер загружает свечи по API биржи.
4. Data Normalization Layer преобразует ответ в `OHLCVBar` и `MarketSeries`.
5. Нормализованные данные передаются в аналитический конвейер.

## 7. Правила выбора торговой пары

Для первой версии принимаются следующие правила:

- Основная котируемая валюта: `USDT`.
- Целевая форма пары: `BASE + USDT`.
- Если актив из топа CoinMarketCap не имеет доступной пары вида `BASEUSDT` на выбранной бирже, он исключается из автодобавления.
- Ручное добавление пары пользователем имеет приоритет над автонабором рынка.

Примеры:

- `BTC` -> `BTCUSDT`
- `ETH` -> `ETHUSDT`
- `TON` -> `TONUSDT`, если такая пара существует у провайдера свечей

## 8. Правила фильтрации инструментов

На уровне проектирования вводятся такие фильтры:

- Исключать стейблкоины.
- Исключать активы без пары на выбранной бирже.
- Исключать активы с пустыми или недостаточными свечными данными.
- Исключать дублирующиеся инструменты после маппинга.

Для первой версии стейблкоинами считаются активы, привязанные к фиатной стоимости и используемые как расчетные токены.

## 9. Нормализованный контракт для свечей

Каждая свеча внутри приложения должна содержать:

- `open_time`
- `close_time`
- `open`
- `high`
- `low`
- `close`
- `volume`
- `symbol`
- `timeframe`

Требования:

- Свечи должны идти в строгом временном порядке.
- Не должно быть дубликатов по времени открытия.
- Все числовые поля должны быть приведены к числовому формату до передачи в доменную логику.
- Неполная текущая свеча не должна использоваться в финальном анализе, если это может искажать структуру.

## 10. Контракт для таймфреймов

Внутри приложения используется канонический набор значений:

- `5m`
- `15m`
- `1h`
- `4h`
- `1d`

Правила:

- Внешний формат провайдера должен быть преобразован к этим значениям.
- Если таймфрейм не поддерживается провайдером, запрос должен завершаться контролируемой ошибкой.
- Таймфрейм по умолчанию берется из `AppSettings`.
- Начальный таймфрейм при старте бота для первой версии: `5m`.

## 11. Ошибки и лимиты API

### 11.1 Категории ошибок

- Ошибка сети.
- Таймаут запроса.
- Превышение rate limit.
- Неверная или недоступная пара.
- Пустой ответ API.
- Некорректный формат ответа.
- Недостаточная глубина исторических данных.

### 11.2 Правила обработки

- Ошибки одного инструмента не должны останавливать общий цикл мониторинга.
- Ошибки должны возвращаться в структурированном виде.
- Для временных ошибок допускаются повторные попытки.
- Для устойчивых ошибок пара может временно пропускаться до следующего цикла.
- При систематической ошибке провайдера Monitoring Coordinator должен уменьшать нагрузку и логировать деградацию.

### 11.3 Повторные попытки

- Для сетевых ошибок и таймаутов используются ограниченные retry.
- Для ошибок rate limit используется отложенный повтор с увеличением паузы.
- Для ошибки несуществующей пары повторная попытка в рамках текущего цикла не нужна.

## 12. Минимальная стратегия rate limiting

Для первой версии слой данных должен поддерживать:

- ограничение частоты запросов к CoinMarketCap;
- ограничение частоты запросов к бирже свечей;
- очередь запросов вместо неконтролируемого параллелизма;
- короткий backoff при ответе провайдера о перегрузке.

## 13. Кэширование

Для первой версии допускается только легкий прикладной кэш:

- кэш списка доступных торговых пар биржи;
- кэш списка стейблкоинов;
- краткоживущий кэш недавних рыночных запросов в пределах одного цикла.

Не требуется:

- долгосрочное хранение полного архива свечей;
- сложный распределенный кэш.

## 14. Поведение при недостатке данных

Если данных недостаточно для анализа:

- пара не передается в Wave Analysis Service;
- причина фиксируется как `insufficient_history`;
- пользователь при ручной проверке получает понятное объяснение;
- в фоновом мониторинге отправка сигнала не происходит.

## 15. Конфигурация слоя данных

Обязательные параметры окружения:

- `MARKET_UNIVERSE_PROVIDER`
- `MARKET_DATA_PROVIDER`
- `EXCHANGE`

Рекомендуемые параметры окружения:

- `CMC_API_KEY`
- `DEFAULT_QUOTE_ASSET`
- `MAX_AUTO_PAIRS`
- `REQUEST_TIMEOUT_SECONDS`
- `RETRY_COUNT`
- `RATE_LIMIT_DELAY_MS`

Требования:

- Конфигурация читается из окружения.
- Настройки совместимы с локальным запуском и Docker.
- Секреты не хранятся в коде и не попадают в логирование.

## 16. Docker-совместимость

- Ключи API и параметры провайдеров передаются через переменные окружения контейнера.
- Таймауты, retry и лимиты должны управляться без пересборки образа.
- Логи слоя данных должны быть доступны через стандартный вывод контейнера.
- Путь постоянного хранения ошибок и состояния должен указывать на внешний volume, если хранение файловое.

## 17. Что входит в первую реализацию

- CoinMarketCap как источник топовых монет.
- Один биржевой провайдер свечей.
- Маппинг `BASE -> BASEUSDT`.
- Нормализация свечей в `MarketSeries`.
- Базовая обработка ошибок, retry и rate limit.

## 18. Что не входит в шаг 4

- Поддержка нескольких бирж одновременно.
- Арбитраж между провайдерами.
- Исторический архив свечей.
- Умный fallback между несколькими источниками данных.

## 19. Результат шага 4

- Зафиксирован контракт слоя получения рыночных данных.
- Разделены роли CoinMarketCap, биржевого провайдера свечей, маппинга тикеров и нормализации.
- Определены правила фильтрации инструментов, обработки ошибок, retry и rate limit.
- Следующий шаг можно посвятить базовому движку поиска волн, используя уже понятный входной поток данных.
