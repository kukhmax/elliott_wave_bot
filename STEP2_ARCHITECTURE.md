# Шаг 2. Архитектура модулей Elliott Bot

## 1. Цель шага

- Определить архитектуру проекта до начала кодирования.
- Разделить систему на независимые модули с понятными зонами ответственности.
- Зафиксировать поток данных от рыночных данных до Telegram-уведомления.

## 2. Архитектурный принцип

- Архитектура строится как последовательный конвейер обработки данных.
- Каждый модуль отвечает только за один класс задач.
- Бизнес-логика анализа волн не должна зависеть от Telegram, конкретной биржи или способа хранения.
- Все внешние интеграции выносятся в отдельные адаптеры.
- Центральная оркестрация управляет сценарием, но не содержит детальной логики анализа.

## 3. Слои системы

### 3.1 Слой интерфейсов

- Отвечает за взаимодействие с пользователем через Telegram.
- Принимает команды, кнопки, ручные запросы и настройки.
- Не принимает решений о валидности волн самостоятельно.

### 3.2 Слой оркестрации

- Управляет жизненным циклом бота.
- Запускает и останавливает мониторинг.
- Планирует сканирование пар.
- Передает задачи в аналитические и инфраструктурные модули.

### 3.3 Слой доменной логики

- Содержит правила Эллиотта, проверки Фибоначчи и оценку сценариев.
- Не зависит от Telegram, API биржи и способа хранения данных.

### 3.4 Слой инфраструктуры

- Получает рыночные данные из внешних API.
- Хранит настройки, список пар и историю сигналов.
- Строит графики и отправляет уведомления.

## 4. Основные модули

### 4.1 Bot Interface

Назначение:

- Обрабатывать команды и кнопки Telegram.
- Показывать пользователю списки пар, результаты проверки и статусы.

Вход:

- Команды пользователя.
- Данные от Monitoring Coordinator и Notification Service.

Выход:

- Команды в Monitoring Coordinator.
- Запросы в Settings Service.
- Запросы ручной проверки в Manual Check Service.

### 4.2 Monitoring Coordinator

Назначение:

- Управлять фоновым мониторингом.
- Запускать циклы сканирования.
- Формировать очередь пар на обработку.

Вход:

- Команды start, stop, update settings.
- Текущий список отслеживаемых пар.
- Интервал сканирования и режим поиска.

Выход:

- Задачи на загрузку данных в Market Data Service.
- Задачи на анализ в Wave Analysis Service.
- Сигналы в Notification Service.

### 4.3 Market Universe Service

Назначение:

- Получать список топовых монет.
- Отфильтровывать стейблкоины и неподходящие активы.
- Преобразовывать монеты в поддерживаемые торговые пары.

Вход:

- Запрос на инициализацию списка пар.
- Настройки биржи и лимита количества инструментов.

Выход:

- Список валидных торговых пар для мониторинга.

### 4.4 Market Data Service

Назначение:

- Загружать свечи по нужной паре и таймфрейму.
- Нормализовать формат рыночных данных.
- Контролировать ошибки API, пустые ответы и лимиты.

Вход:

- Пара.
- Таймфрейм.
- Глубина загрузки истории.

Выход:

- Нормализованный набор свечей.
- Статус ошибки загрузки, если данные недоступны.

### 4.5 Extremum Detection Service

Назначение:

- Выделять локальные максимумы и минимумы.
- Подготавливать опорные точки-кандидаты для волнового анализа.

Вход:

- Массив свечей.
- Параметры чувствительности поиска экстремумов.

Выход:

- Набор локальных экстремумов.
- Подготовленные последовательности точек для дальнейшего анализа.

### 4.6 Wave Analysis Service

Назначение:

- Строить кандидатные 5-волновые сценарии.
- Проверять базовую структуру и направленность волн.
- Передавать кандидаты в модуль строгой валидации.

Вход:

- Экстремумы.
- Правила из спецификации шагa 1.

Выход:

- Список кандидатных сценариев.
- Причины первичного отклонения слабых сценариев.

### 4.7 Elliott Validation Service

Назначение:

- Проверять жесткие правила классического импульса.
- Учитывать исключения для диагоналей.
- Проверять пропорции, чередование и подтверждающие признаки.
- Присваивать итоговый статус: подтвержденный, вероятный, отклоненный.

Вход:

- Кандидатный сценарий.
- Формальные правила из функциональной спецификации.

Выход:

- Валидированный сценарий.
- Список выполненных и нарушенных правил.
- Уровень достоверности.

### 4.8 Signal Registry Service

Назначение:

- Хранить уже отправленные сигналы.
- Блокировать повторные уведомления по одной и той же структуре.
- Сравнивать новый сценарий с ранее отправленным.

Вход:

- Валидированный сценарий.
- История сигналов.

Выход:

- Решение: отправлять сигнал или пропустить как дубликат.

### 4.9 Chart Rendering Service

Назначение:

- Строить график по свечам и опорным точкам P0-P5.
- Размечать волны и добавлять краткую подпись сценария.

Вход:

- Свечи.
- Валидированный сценарий.

Выход:

- Файл графика для отправки пользователю.

### 4.10 Notification Service

Назначение:

- Отправлять уведомления в Telegram.
- Формировать единый текст сигнала для автоматического мониторинга и ручной проверки.

Вход:

- Итоговый сценарий.
- График.
- Канал отправки и контекст команды.

Выход:

- Отправленное сообщение или статус ошибки отправки.

### 4.11 Settings Service

Назначение:

- Управлять настройками по умолчанию.
- Хранить интервал сканирования, таймфрейм, режим поиска и параметры чувствительности.

Вход:

- Команды изменения настроек.
- Значения по умолчанию при запуске.

Выход:

- Актуальные настройки для всех остальных модулей.

### 4.12 Watchlist Service

Назначение:

- Хранить список отслеживаемых пар.
- Поддерживать добавление, удаление и смену таймфрейма для каждой пары.

Вход:

- Команды пользователя.
- Список пар из Market Universe Service.

Выход:

- Актуальный watchlist для Monitoring Coordinator.

### 4.13 Manual Check Service

Назначение:

- Выполнять разовую проверку пары по запросу пользователя.
- Использовать тот же аналитический конвейер, что и фоновый мониторинг.

Вход:

- Пара.
- Таймфрейм.
- Текущие настройки анализа.

Выход:

- Итог ручной проверки.
- Причина отклонения, если структура не подтверждена.

### 4.14 Persistence Service

Назначение:

- Сохранять настройки, watchlist, историю сигналов и служебное состояние.
- Восстанавливать рабочее состояние после перезапуска.

Вход:

- Данные от Settings Service, Watchlist Service и Signal Registry Service.

Выход:

- Сохраненное и восстановленное состояние приложения.

### 4.15 Container Runtime Layer

Назначение:

- Обеспечивать стандартный запуск бота через Docker.
- Изолировать окружение исполнения от локальной машины.
- Передавать приложению конфигурацию через переменные окружения и подключенные тома.

Вход:

- Docker image приложения.
- Переменные окружения.
- Конфигурационные и data-volume директории.

Выход:

- Контейнеризированный процесс бота с воспроизводимым окружением запуска.

## 5. Главные потоки данных

### 5.1 Поток запуска мониторинга

1. Bot Interface получает команду запуска.
2. Monitoring Coordinator запрашивает инструменты у Market Universe Service.
3. Market Universe Service возвращает список торговых пар.
4. Watchlist Service сохраняет стартовый список отслеживания.
5. Monitoring Coordinator запускает цикл сканирования.

### 5.2 Поток фонового анализа пары

1. Monitoring Coordinator берет пару из watchlist.
2. Market Data Service загружает свечи.
3. Extremum Detection Service выделяет экстремумы.
4. Wave Analysis Service строит кандидатов.
5. Elliott Validation Service валидирует сценарии.
6. Signal Registry Service проверяет дубликаты.
7. Chart Rendering Service строит график.
8. Notification Service отправляет сигнал в Telegram.

### 5.3 Поток ручной проверки

1. Bot Interface получает пару и таймфрейм от пользователя.
2. Manual Check Service запускает тот же аналитический конвейер.
3. Notification Service возвращает найденный сценарий или причину отклонения.

### 5.4 Поток изменения настроек

1. Bot Interface получает новые значения.
2. Settings Service валидирует и сохраняет их.
3. Monitoring Coordinator использует обновленные значения в следующих циклах сканирования.

## 6. Границы ответственности

- Bot Interface не анализирует волны и не работает с API рынка напрямую.
- Market Data Service не интерпретирует волны и не отправляет сообщения.
- Wave Analysis Service не хранит состояние пользователя и не решает вопросы Telegram.
- Elliott Validation Service не строит графики и не управляет очередью пар.
- Monitoring Coordinator не содержит детальных правил Эллиотта.

## 7. Модель взаимодействия модулей

- Все пользовательские команды сначала попадают в Bot Interface.
- Все операции мониторинга инициируются через Monitoring Coordinator.
- Все аналитические решения принимаются только внутри Wave Analysis Service и Elliott Validation Service.
- Все внешние побочные эффекты проходят через инфраструктурные сервисы: API, хранилище, графики, Telegram.
- Docker-слой не меняет бизнес-логику модулей, а только предоставляет единый способ сборки и запуска приложения.

## 8. Запуск через Docker

### 8.1 Цель контейнеризации

- Обеспечить одинаковый запуск бота на локальной машине, сервере и тестовом окружении.
- Упростить установку зависимостей и запуск без ручной настройки системы.
- Подготовить проект к дальнейшему развертыванию на VPS или другом хостинге.

### 8.2 Что должно входить в Docker-слой

- Dockerfile для сборки образа приложения.
- `.dockerignore` для исключения лишних файлов из контекста сборки.
- Конфигурация запуска контейнера с переменными окружения.
- Подключаемый volume для сохранения состояния, если хранение будет файловым.

### 8.3 Что контейнер должен обеспечивать

- Запуск единственной точки входа приложения.
- Доступ к конфигурации Telegram и внешних API через переменные окружения.
- Сохранение данных watchlist, настроек и истории сигналов вне контейнера, если контейнер пересоздается.
- Возможность перезапуска без повторной ручной настройки окружения.

### 8.4 Архитектурное влияние Docker

- Docker добавляется как способ развертывания, а не как отдельный доменный модуль.
- Persistence Service должен поддерживать путь хранения, пригодный для внешнего volume.
- Конфигурация приложения должна читаться из окружения, а не быть жестко зашитой в код.
- Точка входа приложения должна запускать Monitoring Coordinator и инфраструктурные сервисы без зависимости от локального режима разработки.

## 9. Минимальная структура проекта

- `app` — точка входа и запуск приложения.
- `interfaces` — Telegram-команды и обработчики пользовательских действий.
- `orchestration` — координаторы мониторинга и ручных проверок.
- `domain` — модели волн, правила Эллиотта, валидация, scoring.
- `services` — сценарные сервисы прикладного уровня.
- `integrations` — клиенты внешних API.
- `storage` — сохранение настроек, watchlist и истории сигналов.
- `rendering` — построение графиков.
- `deploy` — Dockerfile, конфигурация запуска и связанные артефакты развертывания.
- `shared` — общие типы, конфиг и вспомогательные утилиты.

- Архитектура разбита на отдельные модули с понятными входами и выходами.
- Определены главные потоки данных и границы ответственности.
- В архитектуру добавлен стандартный запуск бота через Docker без смешивания с доменной логикой.
